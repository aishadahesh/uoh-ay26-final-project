# PRD: Interactive Play Mode (Agent vs Agent / Human vs Agent / Human vs Human)

**Mechanism owner:** `src/police_thief/domain/interactive_match.py`, `domain/board.py::Board.legal_moves`, `gui/board_canvas.py` (click-to-move extension), `gui/mode_select.py`, `gui/play_app.py`; CLI wiring in `main.py`'s `play` subcommand
**Parent document:** `docs/PRD.md` (see §7, "Per-Mechanism PRD Documents")
**Corresponds to:** no section of `docs/tasks.md` — this is a deliberate addition beyond the rulebook's own scope, built at direct user request after comparing this project's GUI against the user's own separate prior course project (`github.com/aishadahesh/uoh-ay26-mcp-ai-cops-and-robbers`), which has a mode-selectable, human-playable GUI (`cop_agent_robber_user` / `cop_user_robber_agent` / `cop_user_robber_user` / two-AI modes, a directional move pad, click-to-move, and barrier placement).

## 1. Description & Theoretical Background

Every mechanism in Chapters 1–11 is built around **autonomous agents**: `BrainBase` (Chapter 6) has no parameter through which a human could inject a move, and the Live GUI (Chapter 7) is a pure observation window onto an agent's own belief state. That is correct and unchanged for the competitive, submitted system — but it also means there was no way to actually *play* the game as a human, which is a genuinely different, complementary need: a practice/demo tool, not a competitive-match interface.

This mechanism adds that capability without touching anything already built: a new, framework-agnostic `InteractiveMatch` engine (mirroring the "pure logic, thin GUI" split established since Chapter 7's `LiveViewModel`) drives one playable match under any of four selectable modes, and a new Tkinter window (`PlayApp`) lets a human act via a move pad, board clicks, or barrier placement whenever it's their turn. Chapter 6's real `ManhattanHeuristicBrain`/heuristic machinery is reused for any agent-controlled side (via `greedy_manhattan_move`, the same function Chapter 6 and the earlier `demo` command already use) — this is not a separate, parallel AI implementation.

The one deliberately interesting theoretical wrinkle: in any mode with at least one agent side, the human player sees **exactly what an agent would see** — their own position and a belief heatmap built from real scent (Chapter 4), never the opponent's true position. Playing the human side is, quite literally, playing the same partial-observability problem Chapter 1's Dec-POMDP formalism describes — the human has to hunt or evade using the same fog-of-war an autonomous agent operates under.

## 2. Specific Requirements, Input/Output, Performance Metrics

| Requirement | Implementation | Expected input → output |
|---|---|---|
| Four selectable modes | `GameMode` (`AGENT_VS_AGENT`, `HUMAN_COP_VS_AGENT`, `AGENT_VS_HUMAN_THIEF`, `HUMAN_VS_HUMAN`), `controller_for(mode, role)` | each mode maps each role to exactly `PlayerType.AGENT` or `PlayerType.HUMAN` |
| Legal-move computation | `Board.legal_moves(pos) -> dict[Move, Position]` | reuses `apply_move`'s own validation; a boxed-in position still legally offers `STAY` |
| Turn arbitration | `InteractiveMatch.is_human_turn()` | `True` only when the *current* role's controller (per the active mode) is human |
| Agent move computation | `InteractiveMatch.agent_move()` | delegates to Chapter 6's `greedy_manhattan_move`, chasing the belief's `arg_max()` for the cop, fleeing it for the thief |
| Human move/barrier application | `InteractiveMatch.apply_move(move)` / `.place_barrier(target)` | updates position/scent/belief, checks capture/boxed-in/max-moves, advances the turn; raises `MoveRejectedError`/`ValueError` for an illegal action, `RuntimeError` if the match already ended |
| Local Truth preserved for any agent-involving mode | `InteractiveMatch.visible_view_for_current() -> VisibleView` | `belief` populated, `opponent_position` is `None` |
| The one deliberate, user-confirmed exception | same method, `GameMode.HUMAN_VS_HUMAN` | `belief` is `None`, `opponent_position` populated — both true positions shown on one shared board, since both humans share one physical screen by construction |
| Click-to-move | `BoardCanvas.cell_from_pixel`/`set_click_handler`/`highlight_legal_cells` | a click inside a highlighted legal cell fires the bound callback with `(row, col)`; a click elsewhere, or with no handler set, is a no-op |
| Mode selection UI | `ModeSelectDialog` | a modal radio-button dialog; `show()` blocks until Start is clicked, returning the chosen `GameMode` |
| Interactive window | `PlayApp` | move-pad buttons enabled/disabled per legality and turn; a Place-Barrier toggle (cop-only); agent turns auto-advance via `after(500ms, ...)`; match end shows a result message and disables all controls |
| Performance | Turn advancement is O(board size) belief/scent work plus negligible Tkinter redraw; agent turns are deliberately delayed 500ms for human-readable pacing, not a performance necessity | full 434-test suite runs in ~17s |

## 3. Constraints, Limitations, Alternatives Considered & Rationale

- **Scope boundary — explicitly beyond the rulebook.** No section of `docs/tasks.md` asks for a human-playable mode; `BrainBase`'s entire design (Chapter 6) assumes autonomous decision-making. This mechanism is additive and optional — nothing about the competitive match path (`Orchestrator`, `serve`, the crypto/commit-reveal protocol) changes at all. `InteractiveMatch` is a new, separate engine; it does not touch `run_local_match`, `Orchestrator`, or any Chapter 8/9 code.
- **Design choice — orthogonal-only movement, not the reference project's 8-directional set.** The prior project this was modeled on supports diagonal movement (`Move.NW`/`NE`/`SW`/`SE`) and a keyboard-driven 9-button pad. This project's Sec. 3.3.1 makes diagonal movement `[FORBIDDEN]` — a hard mandatory rule, not a style choice — so the move pad here has exactly the five moves `Board`'s own `Move` enum already defines (N/S/E/W/STAY), and `legal_moves()` reuses the exact same `apply_move` validation every other chapter already relies on. No new movement rule was invented to match the reference project's richer-looking pad.
- **Design choice — the Human-vs-Human shared-board exception was a deliberate, explicit user decision, not a default.** Two options were presented: (a) preserve Local Truth even in hotseat mode by showing only the current mover's own belief-based view each turn (a "pass and play" style), or (b) show both true positions on one board at all times, matching the reference project's and most hotseat board games' familiar convention. The user chose (b) — `VisibleView`'s doc comment and this PRD both record that choice explicitly, scoped *only* to `GameMode.HUMAN_VS_HUMAN`; every other mode (including a human playing against an agent) still gets the strict, tested Local Truth guarantee Chapter 7 established.
- **Design choice — reusing `build_live_view_model` for agent-involving rendering, rather than a second belief-to-color implementation.** `PlayApp._render()` calls the exact Chapter 7 function for any mode with a belief to show, so the same cell-coloring/barrier-rendering logic that's already tested there is not duplicated a second time — the same DRY discipline behind the `BoardCanvas` extraction itself.
- **Design choice — barrier placement finally has a real caller.** Every prior chapter noted that `Board.place_barrier`/the barrier-declaration crypto primitive were correct in isolation but never actually exercised by any strategy (no shipped brain places barriers). `PlayApp`'s barrier button is the first real code path that ever calls `place_barrier` from an interactive session — a genuine, if informal, exercise of previously untested-in-practice game mechanics.
- **A real Tkinter testing limitation found and worked around:** `canvas.event_generate("<Button-1>", ...)` does not reliably fire a bound click callback on a withdrawn/headless window in this environment (verified empirically before trusting it). Tests instead call the bound handler (`BoardCanvas._on_click`) directly with a minimal fake event object exposing `.x`/`.y` — the same "real object, direct method call instead of full OS event simulation" pattern already used for `tk.Button.invoke()` throughout this project's test suite.
- **A real bug-shaped race found and closed, same discipline as Chapter 8's stale-callback test and the Chapter 7 GUI upgrade's stale-tick test:** a scheduled `after(500, self._agent_turn)` call is never cancelled if the match ends by some other means before it fires (e.g., a human captured the opponent on their own turn immediately before the agent's scheduled turn would have run). `_agent_turn` guards against this (`if self.match.is_finished: return`), verified by a dedicated test simulating exactly that race rather than left untested as "shouldn't happen in practice."
- **Limitation — no networked/multi-machine interactive play.** This mode is single-process only, matching the existing `demo` command's scope; two humans playing from separate machines, or a human playing against a networked opponent process, would require wiring `InteractiveMatch` into the Orchestrator/commit-reveal protocol, which is out of scope for what was asked (a local, practice-oriented "premium GUI" experience).

## 4. Success Criteria & Test Scenarios

All satisfied, per `tests/unit/test_board.py` (new `legal_moves` tests), `test_interactive_match.py`, `test_board_canvas.py` (click/highlight additions), `test_mode_select.py`, and `test_play_app.py` (68 new tests directly on this mechanism; 434 tests total project-wide, 99%+ coverage, zero ruff violations, 100% coverage on every file this mechanism touches):

1. `Board.legal_moves()` correctly includes all five moves in open space, excludes off-board directions at a corner, excludes a blocked neighbor, and still offers `STAY` even when fully boxed in.
2. `controller_for()` correctly maps every role in every one of the four modes to the intended `PlayerType`; `is_human_turn()` tracks the current role correctly as turns alternate.
3. `visible_view_for_current()` returns a populated `belief` and `opponent_position=None` in every agent-involving mode, and the reverse (no belief, both true positions) only in `HUMAN_VS_HUMAN` mode.
4. `apply_move`/`place_barrier` correctly update position/scent/belief, detect a direct-move capture, a barrier-placement capture, a boxed-in capture triggered by an unrelated action, and a max-moves survival outcome; both raise cleanly once the match has already ended, and `place_barrier` raises when attempted out of turn or by the wrong role.
5. `BoardCanvas`'s click-to-cell translation is correct in-bounds and returns `None` out-of-bounds; the click callback fires with the right cell, is silenced when disabled, and ignores out-of-bounds clicks; legal-cell highlighting outlines exactly the given set and correctly clears previous highlights not in a new one.
6. `ModeSelectDialog` defaults to Agent vs Agent, correctly captures the selected radio button on Start, destroys itself, and `show()` — genuinely exercising its blocking `wait_window()` call via a scheduled `after()` click rather than skipping it — returns the chosen mode.
7. `PlayApp` correctly enables only the legal move buttons and the click handler on a human turn; disables every control and schedules the agent on an agent turn; applies a move/barrier from a button, a board click, or the pad correctly, including capture-via-click and boxed-in-via-unrelated-action; is a no-op (never crashes) for an illegal move, an illegal barrier target, a click when it isn't the human's turn, or a stale scheduled agent turn firing after the match already ended some other way; and shows a real (mocked, non-blocking in tests) result dialog and disables all controls once the match ends.
8. The full `play` CLI command runs end-to-end headlessly (mode selection → match construction → `PlayApp` wiring → `root.mainloop()`) without crashing.
