## 1. Description & Theoretical Background

The rulebook's central architectural break from a conventional client-server game is Sec. 2.1's **full decentralization**: there is no central game server arbitrating turns, no authoritative third party holding the "real" board state, and no asymmetry between the two sides. Each of the two agents (cop, thief) is simultaneously an MCP **server** (exposing tools the opponent calls into) and an MCP **client** (calling the opponent's own exposed tools) — a fully symmetric relationship, never a "strong side" driving a "weak side."

FastMCP (a Python implementation of the Model Context Protocol) is the chosen transport specifically because it already solves the boring, error-prone part of this problem: given a plain Python function, it derives a JSON schema from the function's signature, validates every inbound call against that schema, and rejects a malformed call (missing field, wrong type, unexpected extra field) with a clean, structured error before any of this project's own code runs at all. That is a meaningful amount of correctness bought for free, and Chapter 2 deliberately does not re-implement any of it.

Sec. 2.3.4's **Total Separation of Working Environments** is the second load-bearing rule this layer exists to satisfy: cop and thief must never share runtime memory or a config file. This project's chosen realization of that rule (see `docs/PLAN.md` ADR-011) is a single shared Python package with role-differentiated behavior via a `--role` CLI flag and per-role config directories — separation is enforced by *process boundary and directory boundary*, not by maintaining two entirely separate source trees, which would only duplicate code without adding any real isolation guarantee a single package with disciplined config loading doesn't already provide.

## 2. Specific Requirements, Input/Output, Performance Metrics

| Requirement | Implementation | Expected input → output |
|---|---|---|
| Every peer is simultaneously server and client | `build_peer_server`/`run_peer_server` (server half) + `send_move`/`send_move_async` (client half), both present in every process | the same running process both serves `receive_move` and calls the opponent's `receive_move` |
| A tool-call schema derived automatically, not hand-validated | `@mcp.tool(version=TOOL_SCHEMA_VERSION) def receive_move(signed_move: str, signature: str) -> dict` | a call missing a required field or carrying an unexpected extra field → FastMCP's own `ToolError`, never reaching `receive_move`'s body |
| Structural (non-cryptographic) input validation as a second layer | `MoveEnvelope.is_structurally_valid()` — rejects blank/whitespace-only fields | `signed_move=""` or `signature="   "` → `{"accepted": False, "move": None}` |
| A real, addressable network transport, not an in-process stub | `run_peer_server(mcp, host="0.0.0.0", port)` via `mcp.run(transport="http", ...)`; `send_move_async(opponent_url, ...)` via a real `fastmcp.Client(opponent_url)` | a genuine HTTP round trip over an actual OS-bound port — proven, not simulated, in `tests/integration/test_mcp_http_roundtrip.py` |
| Binding for future tunnel exposure | server binds `0.0.0.0`, not `127.0.0.1` | a tunneling tool (ngrok/Localtonet, Sec. 2.3/Stage 5) can forward external traffic to this same bound port without a code change |
| Environment separation: each role loads only its own config | `load_network_config(role, config_root)`, `config_dir_for(role, config_root)` | `role=COP` → reads only `config/cop/game.toml`; never opens `config/thief/`'s directory, even to check it exists |
| A missing/malformed config fails loudly, not silently | `ConfigError` on a missing file or a missing required TOML key | absent `config/cop/game.toml` → `ConfigError` naming the exact missing path |
| Client failures are caught and typed, never a bare stack trace across the network boundary | `PeerClientError`, wrapping any underlying `fastmcp.Client`/`httpx` exception | an unreachable `opponent_url` → `PeerClientError` naming the address, not a raw connection-refused traceback |
| Performance | A single HTTP round trip per move call, no retry/backoff logic at this layer (that is Chapter 8's Deadline Tracker) | the original Chapter 2 suite (20 tests spanning this module) still passes in ~6s when re-run against today's codebase |

## 3. Constraints, Limitations, Alternatives Considered & Rationale

- **Scope boundary — transport, not payload.** `MoveEnvelope`'s `signed_move`/`signature` fields were, and remain in this module, deliberately generic placeholders: real board moves (Chapter 3) and real SHA-256 commitments (Chapter 5) are layered on top by later chapters' callers (Chapter 8's `Orchestrator`, specifically), not by rewriting this module. `mcp_server.py`'s own docstring states this explicitly, and it has held true through every later chapter — Chapter 8's `Orchestrator.run_turn` calls `send_move_async` with a real commitment hash in the same `signed_move: str` parameter this module has exposed unchanged since Chapter 2, exactly as anticipated.
- **Design choice — a single shared package, not two separate repositories, for environment separation (ADR-011).** The rulebook's own Sec. 2.3.4 requires separation of *runtime memory and config*, not separation of *source code repositories* — App. C's submission requirement for two GitHub repos is a submission-packaging concern (Chapter 9/App. C), addressed at submission time, not a constraint on how the code is developed. A single package with `--role`-gated behavior and per-role config directories (`config/cop/`, `config/thief/`) gives the same runtime isolation guarantee (two independent OS processes, two independent config files, zero shared Python objects) with far less duplication risk than hand-maintaining two divergent copies of identical transport code.
- **Design choice — FastMCP's own schema validation is trusted, not duplicated.** `test_receive_move_rejects_missing_required_field`/`test_receive_move_rejects_unexpected_extra_field` (Chapter 2) assert that FastMCP itself does this rejection, deliberately proving the framework's behavior rather than writing this project's own parallel field-checking logic that could drift out of sync with what FastMCP actually enforces.
- **Design choice — `mcp_client.py` implements no retry or timeout-beyond-a-single-call logic.** Sec. 2's client wrapper is, by design, transport-only: `send_move_async` either succeeds within its single `timeout` or raises `PeerClientError` once. Bounded retries and heartbeat-based liveness detection are a distinct concern with their own design space (backoff strategy, "how many attempts," "what counts as still alive") that Chapter 8 owns in full (`DeadlineTracker`, `Watchdog`) — folding a half-designed retry loop into the transport wrapper here would have meant redesigning it again once Chapter 8 arrived.
- **Limitation — no tunneling was ever actually configured or tested (Stage 5 / Section G of `docs/TODO.md`).** The server binds `0.0.0.0` specifically so a tunnel *can* be added without a code change, but no `ngrok`/`Localtonet` session was ever run, no public URL was ever issued, and no genuinely cross-machine match was ever executed. This requires an external tool, a real second machine (or a teammate's), and a live network session — none of which are available inside an automated coding session. This is the same category of limitation as Chapter 9's deferred Gmail OAuth setup: a real, external, manual dependency, not a shortcut taken for convenience. **This is flagged as a manual step for you**: install `ngrok` (or `Localtonet`), run `ngrok http <port>` against a running `serve --role cop`/`serve --role thief` process, update the opponent's `config/<role>/game.toml`'s `opponent_url` to the issued public URL, and confirm a `send_move` call succeeds across that tunnel — everything on the code side (the `0.0.0.0` bind, the URL-driven client) is already in place and requires no change.
- **Limitation — no genuine two-process, turn-alternating match over the network yet.** Chapter 2 proved a single request/response round trip over real HTTP; Chapter 8's `Orchestrator` proved one real turn (commit → send → self-verify) against a real FastMCP server. Neither proves a *complete, multi-turn match* between two independent, alternating processes exchanging real board state turn-by-turn — that requires the not-yet-built continuous match loop/two-sided driver already flagged as absent in `docs/PRD_reliability_layer.md` §3. `docs/TODO.md`'s Stage-2 milestone item covering exactly this ("a full two-process match end-to-end... using the Stage-1 placeholder policies over the network") remains honestly unchecked for the same reason.

## 4. Success Criteria & Test Scenarios

All satisfied per `tests/unit/test_mcp_server.py`, `test_mcp_client.py`, `test_config.py`, and `tests/integration/test_mcp_http_roundtrip.py` (20 tests directly on this mechanism, re-confirmed passing against the current codebase in Chapter 10 exactly as they did when written in Chapter 2 — nothing in this module has changed since):

1. `MoveEnvelope.is_structurally_valid()` correctly distinguishes well-formed payloads from blank/whitespace-only fields, in every single-field-blank combination.
2. FastMCP's own schema derivation rejects a `receive_move` call missing a required field, and separately rejects a call carrying an unexpected extra field — proving the framework's validation is actually active, not merely assumed.
3. A **real** HTTP server, bound to an actual OS port in a background thread (not FastMCP's in-memory transport), correctly accepts a well-formed move and correctly reports `accepted: False` for a blank signature — a genuine network round trip, re-verified passing today.
4. `send_move`/`send_move_async` correctly raises `PeerClientError`, naming the unreachable address, when the opponent cannot be reached.
5. `load_network_config` correctly loads a valid `config/<role>/game.toml`, applies the documented default `turn_timeout_seconds` when the key is absent, raises `ConfigError` on a missing file or missing required key, and — critically for the environment-separation rule — never reads the other role's directory even when explicitly checked for it (`test_load_network_config_never_reads_the_other_roles_directory`).
