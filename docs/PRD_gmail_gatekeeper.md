## 1. Description & Theoretical Background

Chapter 9 answers a question none of the earlier chapters needed to ask: what happens once a correct, well-tested agent is released into a live, adversarial, referee-less league against opponents built by other teams? Two distinct problems appear at that point.

First, **scoring fairness across a whole season, not one match.** A single match's score (Chapter 3) says nothing about whether a team is exploring the league or farming one weak rival. The **Diversity Incentive** (Sec. 9.2.1) and the "only one counted game per opponent, ever" rule (Sec. 9.2.2) exist specifically to reward breadth of competition over repetition — the same anti-overfitting instinct that motivates a held-out test set in classical ML, applied to competitive match-making instead.

Second, **automated Gmail reporting is a genuine liability, not a convenience.** Sec. 9.3.1 states it plainly: a bug in the reporting code now has write access to a real email account. A runaway loop could fire thousands of emails a minute, hit Google's `429 Too Many Requests`, or exhaust a daily quota and lock the account entirely. The **Gatekeeper** is three cumulative, independent defense layers against exactly that failure mode — not a single check, because a single check has a single point of failure. This mirrors, at the infrastructure level, the same "defense in depth" principle Chapter 5's commit-reveal protocol applies at the cryptographic level.

## 2. Specific Requirements, Input/Output, Performance Metrics

| Requirement | Implementation | Expected input → output |
|---|---|---|
| Token-Bucket formula `tokens <- min(C, tokens + r*dt), allow iff tokens >= 1` | `TokenBucket(capacity, refill_rate, clock).allow(cost=1.0)` | full bucket → `True` until drained; refills proportionally to elapsed time, capped at capacity |
| Daily quota, the last line of defense | `QuotaManager(daily_threshold, persist_path, today).allow()`/`.record_send()` | count below threshold → allowed; at/above threshold → blocked, even for one more request |
| Quota persists across a process restart | JSON file at `persist_path`, reloaded and day-checked on construction | a fresh `QuotaManager` instance pointed at the same file/day sees the same count |
| Quota resets at the day boundary | `_rolled_over()` compares stored day to `today()` | crossing midnight (an injected `today` callable in tests) resets the count to 0 |
| DOS/Anomaly Detector (circuit breaker) | `AnomalyDetector(max_sends_in_window, window_seconds, clock).record_attempt()`/`.tripped` | a burst exceeding the window's cap trips permanently; spaced-out sends never trip; once tripped, stays locked even after a quiet period |
| Three gates composed, quota-then-anomaly-then-rate-limit order | `Gatekeeper.submit()` | a send that clears all three gates succeeds and consumes exactly one unit from each stateful gate; a blocked gate consumes nothing |
| `429` backoff, bounded, never blind retries | `Http429BackoffPolicy(retry_backoff_seconds, max_retries).backoff_schedule()`, consumed by `send_match_report` | consistent `429`s exhaust the schedule and give up; a `429` followed by success recovers within budget |
| JSON-attachment-only reports (never free text) | `build_report_email(to_addr, subject, json_payload, filename)` | the function's signature only accepts a `dict`, never a free-text body string — structurally impossible to send unparseable prose as the report |
| Base64url encoding for the Gmail API | `encode_for_gmail_api(message) -> str` | a MIME message → a `urlsafe_b64encode`'d string, exactly what `users.messages.send`'s `raw` field requires |
| Four mandatory, namespaced JSON reports | `declaration_filename`/`config_filename`/`log_filename`/`result_filename(game_id[, sub_game_number])` | `G001`, sub-game `1` → `declaration_G001.json`, `config_G001_g01.json`, `log_G001_g01.json`, `result_G001.json` |
| SHA-256 of the match log inside the results report | `sha256_of_log(entries)`, embedded in `MatchResult.log_sha256` | any single-entry change to the log changes the digest |
| All four repo cross-links in the results report | `RepoCrossLinks` (team A/B × cop/thief) | present as a nested object in every `MatchResult` |
| Mutual sign-off precondition | `results_agree(own, opponent) -> bool` | identical game_id/scores/outcome/log hash → `True`; any single mismatch → `False`, must block sending |
| Diversity Incentive & one-counted-game-per-opponent | `LeagueRecord.record_counted_game(opponent_id, own_score, won)` | a win against a new opponent → `own_score + diversity_reward`; a second counted game vs. the same opponent → `LeagueRuleError` |
| Min/max games-per-team caps | `LeagueRecord.has_passed_minimum`, `.can_count_another_game()` | below min → not yet eligible; at max → further counted games rejected |
| False game-count declaration detection | `verify_game_count_declaration(declared, actual) -> bool` | declared count ≠ actual counted games → `False` (grounds for disqualification per Sec. 9.2.5) |
| Tie rule | `apply_tie_rule(own_total, opponent_total, tie_score)` | exact tie → `(tie_score, tie_score)`; otherwise scores pass through unchanged |
| Performance | All Gatekeeper/league logic is in-memory bookkeeping plus one small file read/write (`QuotaManager`); no real network call is made in any test | full 325-test suite runs in ~16s |

## 3. Constraints, Limitations, Alternatives Considered & Rationale

- **Update — the real Gmail OAuth transport is now built (`services/gmail_oauth.py`).** Originally deferred as "cannot be built inside an automated coding session," this was revisited after the user pointed at a real, working OAuth flow from a separate prior course project for this same Google account (`aishadahesh11@gmail.com`) — proof the flow itself is sound and reusable, not something to reinvent from scratch. `gmail_oauth.py` ports that proven pattern (`InstalledAppFlow` + a cached, auto-refreshed `token.json`, matching `get_service()`'s shape from App. A Sec. 12.3.2) with one deliberate correction: it requests only the `gmail.send` scope (App. A Sec. 12.1.3/12.2.3's Least-Privilege mandate), narrower than the prior project's `gmail.modify`. `build_gmail_api_transport()` returns a real `Transport` that plugs directly into the already-tested `send_match_report` pipeline with no change to that pipeline at all. What genuinely still cannot happen inside a coding session is the one-time interactive browser consent itself (`InstalledAppFlow.run_local_server()` opens a real browser against a real Google account) and any real network call to the live API — everything else, including credential-loading edge cases (missing files, a malformed cached token, an expired-but-refreshable token, a fully-expired token with no way to re-authorize, and Gmail's real `HttpError`/429 response shape translated into this project's own typed exceptions), is tested for real using the actual `google-api-python-client`/`google-auth-oauthlib` libraries (the optional `email` dependency group, `uv sync --extra email`) against hand-built fake credentials/service objects — never a live network call. **This remains a manual activation step for you**: place a real `credentials.json` (downloaded from your own Google Cloud project, App. A Step D) at the project root, then call `build_gmail_api_transport(Path("credentials.json"), Path("token.json"))` once — the first call opens the real consent flow and caches `token.json` for every call after that.
- **A real bug found and fixed while building this, same discipline as every prior chapter:** the first version of `load_credentials` let a malformed/corrupted `token.json` propagate a raw `json.JSONDecodeError` instead of this module's own `GmailOAuthError` — caught by actually writing a garbage `token.json` and calling the function before trusting it, then fixed by catching `ValueError` around the parse and re-raising with a clear, actionable message.
- **Design choice — `Gatekeeper.submit()`'s check order (quota → anomaly → rate-limit).** Quota and anomaly checks are free reads; the token bucket's own `allow()` call is not free — it mutates the bucket's remaining tokens on success. Checking the two free, stateless-to-check gates first means a blocked send (on quota or anomaly grounds) never wastes a token it didn't need to spend. Verified by `test_a_blocked_send_never_consumes_quota_or_rate_limit_resources`.
- **Design choice — the Anomaly Detector never self-resets once tripped.** A quiet period after a burst does not prove the underlying bug that caused the burst is fixed — it may simply mean the runaway loop paused. Sec. 9.3.12's own framing ("prevents exhausting the account's computational resources by an unresponsive code path") implies the lock should require deliberate human intervention (a restart), not silently heal itself. Verified by `test_once_tripped_the_breaker_stays_locked_even_after_a_quiet_period`.
- **Design choice — reusing Chapter 5's `canonical_json`/SHA-256 for the log hash and Chapter 7's `save_log`/`load_log` verbatim for the log report file, rather than inventing new serialization.** Sec. 9.3.19 itself describes the log file as "for cryptographic audit in a replay simulator" — that is precisely the file Chapter 7's Replay Viewer already reads. Re-implementing it here would risk silent format drift between what Chapter 9 writes and what Chapter 7 reads; `match_reports.py`'s own comment block documents this explicitly rather than duplicating the function.
- **Design choice — the Diversity Incentive collapses cleanly with the one-counted-game rule.** Sec. 9.2.1 ("victory against an opponent not already beaten") and Sec. 9.2.2 ("only one game per opponent counts") read as two separate rules, but because a counted game against a given opponent can only ever happen once, "not already beaten" and "this, your only counted game against them" are the same condition. `LeagueRecord.record_counted_game` implements this as one unified operation rather than two separately-tracked concepts, avoiding a redundant, potentially-inconsistent "already beaten" set alongside the "already counted" set.
- **Design choice — `LeagueRuleError` raises rather than silently ignoring rule violations.** A second counted game against an already-counted opponent, or a counted game beyond `max_games_per_team`, are `[MUST]`-level rulebook violations (Sec. 9.2.2, 9.2.6) — silently dropping the extra game (returning `None` or `0`) would hide a bug in the calling code that scheduled an illegal match in the first place. Raising forces the caller to notice, matching this project's established preference for loud, typed exceptions (`GameConfigError`, `ReplayLogError`, `MatchReportError`) over silent no-ops.
- **Limitation — the 429 backoff schedule is flat, not exponential.** Sec. 9.3.10-9.3.14 specify only a fixed `retry_backoff_seconds`/`max_retries` pair (Table 19's minimums), not an exponential-backoff formula. `Http429BackoffPolicy.backoff_schedule()` therefore returns `[retry_backoff_seconds] * max_retries` — satisfying the literal mandatory minimum rather than inventing an unspecified escalation curve.
- **Limitation — `config/game.toml`'s `[email]` section is documented but commented out** in both `config/cop/game.toml` and `config/thief/game.toml`, since its `recipient`/`mode` fields have no real consumer yet (no live match loop calls `send_match_report` automatically) and wiring it in now would mean guessing at that call site's shape before it exists — the same reasoning already applied to Chapter 8's deferred `response_timeout_sec`/`watchdog_timeout_sec` config wiring.
- **Update — `config/game.json`'s missing `network_and_league`/`rate_limiter_gatekeeper` sections are now present.** App. B Sec. 13.3.1's worked example always included these two sections (plus `world`); this project's config only had `board_and_agents`/`movement_and_barriers`/`scoring`/`pheromones` until now — a real gap, not a deliberate omission. `shared/game_config.py::MatchParameters` gained `world`/`network_league`/`rate_limiter` fields (typed `WorldConfig`/`NetworkLeagueConfig`/`RateLimiterConfig`), loaded and validated the same way as every other section: the four Table-18 FIXED fields (`num_games`, `diversity_reward`, `min_games_to_pass`, `max_games_per_team`) are checked for exact equality (mirroring Chapter 4's `_validate_fixed_scent_config`), and the five Table-19 rate-limiter fields are checked against their mandatory floors (mirroring the existing `grid_size`/`max_barriers` floor checks). This makes the values *loadable and validated*, closing the config-schema half of `docs/TODO.md` T0545/T0565; *wiring* them into `Gatekeeper`/`DeadlineTracker`/`Watchdog`/`LeagueRecord` at runtime still has no call site (no live match loop exists yet), so that half of T0529/T0535/T0566/T0471 remains honestly open.
- **Update — `world.hint_max_words` now flows into `TemplateHintProvider`** (Chapter 6) as an optional, defaulted constructor parameter, rather than the hardcoded `MAX_HINT_WORDS` constant it silently used before. `world.map_area` is loaded and available but deliberately left unused by the template phrases themselves — App. F Table 14 allows an empty string, and the existing direction-only phrasing needs no arena theme to satisfy Sec. 6.4's requirements; wiring it in would be inventing unrequested behavior, not closing a gap.
- **Limitation — no live-match-triggered report sending.** `send_match_report`, the four JSON builders, and `LeagueRecord` are all correct and tested in isolation, but nothing yet calls them automatically at the end of a real match — that requires the Orchestrator (Chapter 8) to be extended with an end-of-match hook, which is a natural next integration point once a full multi-turn match loop exists (see `docs/PRD_reliability_layer.md` §3's own note that no such loop exists yet either).

## 4. Success Criteria & Test Scenarios

All satisfied, per `tests/unit/test_token_bucket.py`, `test_quota_manager.py`, `test_anomaly_detector.py`, `test_gatekeeper.py`, `test_match_reports.py`, `test_gmail_report_sender.py`, `test_gmail_oauth.py`, `test_league.py`, and `test_game_config.py` (67 new/updated tests directly on this mechanism; 380 tests total project-wide, 99.58% coverage, zero ruff violations, 90%+ on every file this chapter touches):

1. The Token Bucket starts full, spends exactly one token per `allow()`, refuses once empty, and refills proportionally to elapsed time without ever exceeding capacity — reproducing Sec. 9.3.10's formula exactly via a fake, injectable clock (no real sleeping in any test).
2. The Quota Manager blocks once the daily threshold is reached (even for a single additional request), survives a simulated process restart via its persisted JSON file, and resets correctly at a simulated day boundary.
3. The Anomaly Detector allows a normal send pattern, trips on a simulated infinite-loop burst, never trips on legitimately spaced-out sends, and — once tripped — stays locked even through a long subsequent quiet period.
4. The composed `Gatekeeper` passes a report cleanly under normal load, blocks correctly (with the specific, correct reason) at each of the three stages under simulated abuse, and never consumes quota/rate-limit resources on a send blocked before it reached them.
5. `build_report_email` produces a real, parseable MIME message with the JSON payload as a genuine attachment (not inline text) and a fixed placeholder body; `encode_for_gmail_api` produces valid base64url output decodable back to the original MIME bytes.
6. `send_match_report`, driven entirely by an injectable fake `transport` (never a real Gmail call): succeeds on the first attempt; recovers from a `429` within the retry budget; correctly exhausts the retry budget and reports failure on a persistent `429`; and stops immediately, without retrying, on a non-`429` hard failure (e.g. an authentication error) — proving retries are reserved specifically for the temporary-failure case the rulebook calls out.
7. All four JSON report builders round-trip correctly through real files, including the declaration report's deeply-nested `SignedStep0` → `Step0Declaration` → `HardwareSpec` chain (Chapter 5); filenames are correctly namespaced by `game_id`/sub-game number; `sha256_of_log` is stable for identical logs and changes on any single tampered entry; `results_agree` correctly detects both full agreement and a single-field score disagreement between two independently-built reports.
8. `LeagueRecord` awards the full Diversity Incentive on a win against a new opponent and none on a loss; correctly rejects (via `LeagueRuleError`) a second counted game against an already-counted opponent and any counted game beyond the `max_games_per_team` cap; `verify_game_count_declaration` correctly distinguishes an honest declaration from a false one; `apply_tie_rule` correctly awards `tie_score` to both sides on an exact tie and leaves any non-tied score untouched.
9. `gmail_oauth.py`'s credential loading correctly: fails fast (no import needed) when neither `credentials.json` nor `token.json` exists; raises a clear `GmailOAuthError` (not a raw `JSONDecodeError`) on a malformed cached token; short-circuits straight through an already-valid cached token without touching the consent flow; calls `refresh()` on an expired-but-refreshable token; runs the (fully mocked, no real browser) consent flow and persists a fresh token when none is cached; and raises clearly when a token is invalid with no `credentials.json` to re-authorize against. The transport wrapper correctly translates a real `HttpError(429)` into `GmailRateLimitedError` and any other `HttpError` into `GmailSendError`, using the actual `google-api-python-client` exception types, not stand-ins.
10. `shared/game_config.py`'s new `world`/`network_league`/`rate_limiter` sections parse correctly from the real shipped `config/game.json`; the four FIXED `network_and_league` fields reject any deviation; all five `rate_limiter_gatekeeper` fields reject a value below their mandatory floor and accept any value raised above it. `TemplateHintProvider` respects a config-supplied `max_words` override and still defaults to the mandatory 15-word baseline when none is given.
