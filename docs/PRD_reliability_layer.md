**Mechanism owner:** `src/police_thief/services/orchestrator.py`, `services/state_machine.py`, `services/deadline_tracker.py`, `services/watchdog.py`, `services/log_manager.py`

## 1. Description & Theoretical Background

Every prior chapter built a correct, independently-tested primitive — board physics (Ch.3), scent (Ch.4), commit-reveal crypto (Ch.5), strategy/belief (Ch.6), GUI/replay (Ch.7) — but none of them were wired to *run a turn end-to-end over the network*, and nothing enforced *legal sequencing* between them. A referee-less P2P match has two independent failure modes this chapter addresses:

1. **Logical ordering failures.** Nothing before this chapter prevented, say, revealing before committing, or re-entering `COMPUTING_MOVE` mid-verification. A **legal state machine** (a finite automaton with an explicit transition table) makes illegal sequencing a structural impossibility rather than a hoped-for convention — any transition not in the table raises immediately rather than silently corrupting the match.
2. **Liveness failures.** A P2P match with no central referee has no external party to notice a hung peer. A **Deadline Tracker** bounds every individual network wait (turning "wait forever" into "fail after N seconds, retry M times, then declare technical loss"), and a **Watchdog** is the complementary, coarser-grained liveness monitor: instead of bounding one request, it detects when the *whole* turn cadence has gone silent (no heartbeat within a threshold), independent of which specific await hung.

The **Orchestrator** is the architectural answer to "who wires all of this together without every module reaching into every other module": a single gateway class that owns one `run_turn()` method, sequencing the state machine, the Decision Module (Ch.6's `BrainBase`), the crypto layer (Ch.5's `commit`/`verify`), the network layer (Ch.2's `send_move_async`), the Deadline Tracker, the Watchdog, and the Log Manager — while itself containing no decision-making or communication logic of its own. This centralization is what makes rules like "a single Orchestrator is the entry point for all sub-systems" (App. E rule 3) verifiable by inspection rather than by convention.

## 2. Specific Requirements, Input/Output, Performance Metrics

| Requirement | Implementation | Expected input → output |
|---|---|---|
| Full legal state set | `MatchState` StrEnum: `WAITING_FOR_OPPONENT`, `COMPUTING_MOVE`, `COMMITTING`, `AWAITING_REVEAL`, `VERIFYING`, `TECHNICAL_LOSS` | — |
| Explicit transition table, illegal transitions rejected | `MatchStateMachine.transition(target)` | any transition not in `_TRANSITIONS[current]` → `IllegalStateTransitionError`, state unchanged |
| `TECHNICAL_LOSS` reachable from every non-terminal state, itself terminal | `_TRANSITIONS` table | `COMPUTING_MOVE`/`COMMITTING`/`AWAITING_REVEAL`/`VERIFYING` → `TECHNICAL_LOSS` legal; `TECHNICAL_LOSS` → anything illegal |
| Bounded per-request timeout + retry | `DeadlineTracker(timeout_seconds, max_retries).call(make_awaitable)` | fast call → result; consistently slow/failing call → `DeadlineExceededError` after `max_retries + 1` attempts, never an indefinite wait |
| Fresh awaitable per attempt | `make_awaitable` is a zero-arg factory, not a bare coroutine | a coroutine can only be awaited once; the factory is re-invoked on every retry |
| Heartbeat-based liveness monitor | `Watchdog(timeout_seconds, on_timeout, clock).heartbeat()` / `.check()` | heartbeat within threshold → `ALIVE`; elapsed time since last heartbeat exceeds threshold → `SHUTDOWN`, `on_timeout` fires exactly once |
| Injectable clock for deterministic testing | `clock: Callable[[], float] = time.monotonic` | tests use a `FakeClock` — no real `sleep()` needed anywhere in the test suite |
| Single Orchestrator gateway | `Orchestrator.run_turn(board, own_position, belief) -> TurnResult` | drives one full turn: `_decide_move` → `commit` → send over real FastMCP → self-`verify` → log → `WAITING_FOR_OPPONENT`; any failure → `TECHNICAL_LOSS` |
| Real commitment wired into the network envelope | `commitment.h_commit` passed as `send_move_async`'s `signed_move` argument | closes the placeholder documented in Ch.2's `MoveEnvelope` docstring since inception |
| Turn-scoped audit log accumulation | `LogManager.record(entry)` / `.entries` (defensive copy) / `.save(path)` (reuses Ch.7's `save_log`) | entries append in order; the returned list is a copy, mutating it never affects internal state |
| Performance | State machine/deadline/watchdog logic is pure in-memory bookkeeping; the only real I/O is one HTTP round-trip per turn | full 272-test suite (including 5 tests against a real local FastMCP server) runs in ~16s |

## 3. Constraints, Limitations, Alternatives Considered & Rationale

- **Scope boundary — one turn, not a full match loop.** The Orchestrator's `run_turn` drives exactly one commit→verify cycle and returns. No continuous main game loop, no multi-turn match driver, and no two-sided "both peers play against each other" harness exist yet. This is a deliberate stopping point: everything Chapter 8's mandatory rules require (single gateway, legal state machine, Deadline Tracker, Watchdog) is demonstrable and testable one turn at a time against a real network peer, and inventing a full match loop now would mean guessing at requirements (win/loss reporting, round counting, final scoring integration) that belong to later chapters/the league layer (Ch.9-10). `docs/TODO.md` Section J leaves T0521 (full-match cycling), T0538-T0540 (continuous heartbeat cadence, state persistence, resume-after-shutdown), and T0543 (mutual-wait stress test) honestly unchecked for exactly this reason.
- **Design choice — `verify()` self-check immediately after commit, inside the same turn.** Rather than waiting for the opponent's own audit pass, the Orchestrator recomputes `verify(state, move, intent, nonce, h_commit)` on its own commitment before logging it. This is deliberately defensive: `commit()`/`verify()` are deterministic and always agree given matching arguments (which `run_turn` always supplies), so in practice this branch cannot fail — but proving the Orchestrator reacts correctly (raises internally, converts to `TECHNICAL_LOSS`, never silently trusts an unverified commitment) required one `unittest.mock.patch`-based test forcing `verify` to return `False`. This mirrors the same "test the defensive branch via mocking, document why it's unreachable in practice" pattern used for Ch.4's full-decay-rate test and Ch.6's non-Windows RAM-detection tests.
- **Design choice — `DeadlineTracker` takes a factory, not a coroutine.** An `Awaitable` passed by value can only be awaited once; a naive `DeadlineTracker.call(coro)` signature would work for the first attempt and then silently break every retry with a cryptic "cannot reuse already awaited coroutine" error. Requiring `Callable[[], Awaitable[T]]` instead makes retry-safety a type-level guarantee, verified by `test_each_retry_gets_a_fresh_awaitable_not_a_reused_coroutine`.
- **Design choice — `Watchdog` uses an injectable clock, not real `sleep()`.** Every prior chapter's testing philosophy has been "real, not mocked" (real FastMCP servers, real Tkinter widgets, real file round-trips) — but a real Watchdog test would mean sleeping past a real timeout in every test run, which is slow and flaky under CI load variance. An injectable `clock: Callable[[], float]` (defaulting to `time.monotonic` in production, a hand-rolled `FakeClock` in tests) gets deterministic, instant test execution without sacrificing the production code path's use of a genuine monotonic clock — the same principle as Step-0's real-vs-mocked hardware detection in Chapter 5.
- **Limitation — config wiring not yet done (T0529, T0535).** `DeadlineTracker`/`Watchdog` are currently constructed directly with explicit `timeout_seconds`/`max_retries` values by callers (tests, and eventually a real entrypoint). Neither `response_timeout_sec` nor `watchdog_timeout_sec` is yet read automatically out of `config/game.json`. This was deferred rather than guessed at, since the real consumer of that wiring is the not-yet-built match entrypoint/main-loop — wiring config now would mean inventing that call site's shape prematurely.
- **Limitation — no `persist_state()`/`controlled_shutdown()`/resume path (T0533, T0534, T0540).** The Watchdog's `on_timeout` callback is a generic hook (`Callable[[], None]`), proven to fire exactly once even under repeated `check()` calls past the threshold. No concrete state-persistence-to-disk or connection-teardown callback has been implemented and wired through it yet, since there is no live match state loop for it to persist or tear down. `LogManager.save()` (reusing Ch.7's `save_log`) is the only persistence primitive that exists today.
- **Alternative considered — background thread for the Watchdog instead of poll-based `check()`.** Rejected for now: a real background thread would need its own shutdown/join lifecycle and interacts awkwardly with `asyncio`'s single-threaded event loop model that `Orchestrator.run_turn` already uses. A poll-based `check()`, called at defined points (turn start/end), is simpler, fully deterministic under the `FakeClock` testing approach, and sufficient for what Chapter 8's rules require; a real background thread remains a reasonable future upgrade once a continuous main loop exists to host it.
- **Real architectural gap closed — the commitment hash now flows over the wire.** Since Chapter 2, `MoveEnvelope.signed_move` had been documented as "becomes a real SHA-256 commitment in Chapter 5/6" but never actually wired — Chapters 5 and 6 built the crypto and strategy primitives in isolation without a caller to connect them. `Orchestrator.run_turn` is that caller: `commitment.h_commit` (Ch.5) is passed directly as `send_move_async`'s `signed_move` argument (Ch.2), verified empirically against a real local FastMCP server.

## 4. Success Criteria & Test Scenarios

All satisfied, per `tests/unit/test_state_machine.py`, `test_deadline_tracker.py`, `test_watchdog.py`, `test_log_manager.py`, and `tests/integration/test_orchestrator.py` (36 new tests directly on this mechanism; 272 tests total project-wide, 99.77% coverage, zero ruff violations, 100% coverage on every file in this mechanism):

1. The state machine's full legal cycle (`WAITING_FOR_OPPONENT → COMPUTING_MOVE → COMMITTING → AWAITING_REVEAL → VERIFYING → WAITING_FOR_OPPONENT`) succeeds; every illegal transition (5 representative combinations, parametrized) is rejected immediately without mutating state; `TECHNICAL_LOSS` is reachable from all four non-terminal states and is itself terminal (rejects every possible target, including itself).
2. `DeadlineTracker.call` returns normally for a fast awaitable; raises `DeadlineExceededError` after exhausting `max_retries + 1` attempts for a consistently slow or failing one; recovers if a later retry within budget succeeds; a zero-retry tracker makes exactly one attempt; each retry receives a genuinely fresh awaitable (proven via a call-counting factory).
3. `Watchdog.check()` reports `ALIVE` immediately after construction and while within threshold; reports `SHUTDOWN` once elapsed time (via the injectable clock) exceeds the threshold; `heartbeat()` correctly resets the elapsed-time window; `on_timeout` fires exactly once even across repeated post-threshold `check()` calls; `shutdown_triggered` reflects state correctly; a `Watchdog` with no callback doesn't crash.
4. `LogManager` starts empty, records entries in call order, exposes `entries` as a defensive copy (mutating the returned list never affects internal state), and round-trips correctly through `save()`/Ch.7's `load_log()`.
5. The Orchestrator completes a full turn against a **real, separately-threaded local FastMCP server** (not mocked): ends in `WAITING_FOR_OPPONENT`, produces a non-`None` move and commitment hash, logs exactly one verifiable `LogEntry`, and heartbeats the Watchdog without triggering shutdown.
6. The Orchestrator correctly declares `TECHNICAL_LOSS` (move/commitment both `None`, log stays empty, state machine ends terminal) when the opponent is unreachable (`http://127.0.0.1:1/mcp`, connection refused) — proving the failure path converts a real network exception into the correct legal state rather than crashing or hanging.
7. A mocked defensive test proves that if `verify()` ever disagreed with `commit()` (which cannot happen in practice, since both are deterministic and always called with matching arguments), the Orchestrator would still correctly declare `TECHNICAL_LOSS` and log nothing, rather than silently trusting an unverified commitment.
